Why the KV cache? To write each new token the model looks back at every earlier token. Without a cache it re-reads the whole sequence each step — O(N²) work that grows with length. The KV cache stores each token's Key/Value once and reuses it, so each step handles only the new token (O(N)). Run the pipeline with the cache engine set to Naive, then again with KV cache: the red curve climbs, the green stays flat. The gap widens with sequence length and model size.

Physical memory page grid

SIMULATION · vLLM PagedAttention allocator
Why PagedAttention? The KV cache lives in GPU memory and grows with every token and every concurrent user. The naive way reserves max-length memory per request up front — but most requests are short, so most of that sits empty and wasted. Wasted memory = fewer users per GPU. PagedAttention hands out memory in small fixed blocks on demand, like an OS paging RAM. This grid is a faithful simulation of that allocator (real KV byte math for the selected model + block size) — HuggingFace can't run the real kernel, vLLM can.

Quantization

measured on a real weight · AWQ/GPTQ/GGUF modeled
Why quantize & dequantize? Weights are normally 16-bit floats — a 7B model is ~14 GB, and every token must stream all of them out of memory (the decode bottleneck). Storing them in fewer bits makes the model smaller and cheaper to read, for an accuracy cost (the % error below). Dequantization is the reverse step before each multiply: the packed ints are expanded back to float using the stored scale + zero-point. Compress to store, decompress to compute. INT8/4-bit run live above; INT2 (GGUF-style) and the INT4 grouping (AWQ/GPTQ-style) are measured here on a real weight matrix.

Continuous batching & concurrency

throughput · how real servers scale
Why batching? One decode step barely uses the GPU — almost all its time goes to reading the model's weights out of memory, not to math. Decode many requests at once and they share that one weight read — far more tokens for nearly the same time per step. That's continuous batching, the biggest throughput trick in vLLM/TGI. How many you can batch is capped by KV memory — exactly what PagedAttention frees up.

Zero-copy transfers

pinned vs pageable memory
Why zero-copy? Data travels from CPU RAM to GPU VRAM over PCIe. Ordinary (pageable) memory can be moved by the OS, so the GPU can't grab it directly — it forces a slow extra copy through a staging buffer. Pinned memory is locked in place, so the GPU pulls it straight across. Same principle in the decode loop: every .item() sync makes the GPU wait on the CPU. Fewer copies and syncs = more time computing.
