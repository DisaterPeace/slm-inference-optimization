Per-token latency

token # vs ms · runs overlay so you can compare
Why the KV cache? To write each new token the model looks back at every earlier token. Without a cache it re-reads the whole sequence each step — O(N²) work that grows with length. The KV cache stores each token's Key/Value once and reuses it, so each step handles only the new token (O(N)). Run the pipeline with the cache engine set to Naive, then again with KV cache: the red curve climbs, the green stays flat. The gap widens with sequence length and model size.

Quantization tradeoff

size vs quality vs speed
Which quant is the best deal? Lower-bit quants are smaller and cheaper to read, but can lose quality. Pick an engine: HuggingFace/GPU sweeps FP16 / INT8 / 4-bit (3 points — bitsandbytes' only levels), llama.cpp/CPU sweeps the full GGUF ladder (Q8→Q2 — the rich curve). It measures size, perplexity (quality, lower is better) and tok/s, and marks the best size↔quality point (the knee, usually Q4_K_M on GGUF). Slow — loads each level (GGUF also downloads; Phi is large).

Static batching throughput

HuggingFace · GPU · tok/s vs batch size
Why batching? A single decode step barely uses the GPU — almost all its time goes to reading the model's weights out of VRAM. Decode B sequences at once and they share that one weight read, so total tokens/sec climbs with batch size while the time per step stays nearly flat. That shared-read win is the foundation of server throughput. Continuous batching is the next step — a scheduler that swaps finished sequences out mid-flight — and it lives in serving engines (vLLM / TGI / llama-server), not the libraries here. (llama.cpp's Python API is single-stream.)
